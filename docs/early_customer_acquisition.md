# Early Customer Acquisition Strategy

## 目标

TraceMind 早期获客目标不是泛泛获得注册量，而是找到一批已经在用 AI coding agent、产品已经有真实用户、但没有成熟 analytics 能力的团队，让他们在 24 小时内拿到第一份可执行的用户行为诊断。

第一阶段成功标准：

- 20 个 design partner 候选进入高质量沟通。
- 5 个真实项目完成接入。
- 3 个项目在 Codex、Claude Code、Cursor 或类似 agent 中查询过 TraceMind 数据。
- 2 个项目基于 TraceMind 的结论改过产品，并验证行为是否变化。
- 至少 1 个客户愿意付费、公开推荐，或授权匿名案例。

## 核心定位

一句话：

> Let your coding agent understand real user behavior.

中文表达：

> 把真实用户行为接进 Codex、Claude Code、Cursor 等 AI coding agent，直接追问用户为什么流失、为什么不付费、为什么不用新功能。

TraceMind 不应被包装成传统 analytics 看板。早期卖点是：

- 一行接入真实行为数据。
- 自动整理成 agent 可查询的产品线索。
- 让开发者在日常 coding workflow 中直接分析用户行为。
- 用实际行为证据支持产品修改，而不是依赖猜测或手工 dashboard。

## 优先客户画像

### A0 类：非技术/弱技术 vibe coding 独立开发者

这是当前第一优先级。

典型特征：

- 个人或 2-3 人小团队，创始人通常不是专业工程师。
- 使用 Lovable、Base44、Floot、Bolt、Replit、Cursor、Claude Code 等工具把产品做上线。
- 产品已经有公开链接、早期用户、waitlist、Product Hunt / Indie Hackers / Reddit 反馈帖，或正在找前 10 个真实用户。
- 还没有成熟 analytics 能力，也不想维护复杂看板和埋点体系。
- 最关心的问题是：有没有人真的用、卡在哪、为什么不转化、改完有没有变好。

高匹配场景：

- 小型 AI 工具、内容工具、个人效率工具。
- 面向 solopreneur、creator、小商家、coach、consultant 的 SaaS。
- 用 AI app builder 快速做出的 MVP、waitlist app、付费小工具。
- 已经在公开社区请求反馈、找 beta user 或找前 10 个真实用户的项目。

核心痛点：

- AI 已经能帮他们把产品做出来，但产品上线后不知道用户怎么用。
- 不懂或不想维护 Mixpanel、PostHog、GA4 这类体系。
- 每次产品没人用，都只能继续让 AI 改界面、加功能，而不知道真正卡点。
- 需要一个更自然的方式：直接问 AI 用户行为，而不是学习数据分析工具。

### A 类：AI-native 小型产品团队

这是第二优先级。

典型特征：

- 1-20 人团队，创始人或 CTO 亲自写代码。
- 日常使用 Cursor、Claude Code、Codex、Copilot Agent 或类似工具。
- 产品已经上线，有真实用户和真实转化问题。
- Web-first 最优，因为一行脚本能最快接入；Native、React Native、混合应用、小程序、浏览器插件、MCP server 和后端服务是加分项。
- 没有专职数据分析师，也没有稳定维护的事件体系。

高匹配场景：

- AI SaaS。
- Devtool。
- PLG 小型 SaaS。
- AI 生成类工具。
- 面向开发者或创作者的在线工具。

核心痛点：

- 功能发得快，但不知道用户真正怎么用。
- 手工埋点跟不上产品迭代。
- 传统 analytics 看板太重，团队没有时间维护。
- 每次用户流失、转化下降、新功能无人使用，都只能猜。

### B 类：AI 应用工作室和外包型 AI builder

这是第三优先级。

典型特征：

- 一个团队反复交付多个 AI app、SaaS MVP 或内部工具。
- 已经把 AI coding agent 放进日常交付流程。
- 有动力把 TraceMind 作为默认交付件，提升客户项目上线后的可观测性。

价值主张：

> 你交付的不只是功能，还有一个能让客户持续追问真实使用情况的 agent-readable behavior layer。

这类客户的价值在于复用：一个工作室可能带来多个项目。

### C 类：早期 B2B SaaS 和内部工具团队

这是第四优先级。

典型特征：

- 已有少量客户、试用或内部用户。
- 关注激活、留存、功能采用、配置完成率。
- 还没有预算或组织能力搭完整数据体系。

筛选条件：

- 技术负责人愿意亲自接入。
- 接入后能在 1 周内产生真实行为数据。
- 有明确问题要问，例如注册、激活、试用到付费、关键功能采用。

## 暂不优先的客户

早期先避开这些客户：

- 大企业和采购流程重的团队。
- 强合规行业，除非有明确试点边界。
- 已有成熟数据团队、完整 CDP/BI/埋点治理的组织。
- 纯广告投放驱动、只关心归因和买量 ROI 的消费 App。
- 还没有上线、没有真实用户的项目。
- 完全不用 AI coding agent 的团队。

这些客户不是长期没有价值，而是早期销售周期长、接入阻力大，且不容易验证 TraceMind 的核心假设。

### 2. GitHub 高信号项目

用公开仓库找已经 agent 化开发的团队。搜索信号：

- `AGENTS.md`
- `CLAUDE.md`
- `.cursor/`
- `.mcp.json`
- `mcpServers`
- `Cursor`
- `Claude Code`
- `Codex`

再筛选：

- README 有 live demo、官网或产品截图。
- 最近 30 天有活跃提交。
- 项目像真实产品，而不是纯 demo。
- 有 issue、release、docs 或用户反馈痕迹。

外联方式：

- GitHub issue 不适合直接销售，除非项目明确欢迎工具建议。
- 优先通过官网、X、LinkedIn、Product Hunt maker 页面或公开邮箱联系。
- 私信要基于具体项目观察，不要群发模板。

### 3. MCP 生态入口

TraceMind 的差异化在 MCP，因此要进入 MCP 生态分发：

- 官方 MCP 文档和生态入口。
- MCP server directory、registry、awesome list。
- Smithery、PulseMCP 等 MCP 发现渠道。
- Claude Code、Cursor、VS Code、Copilot 相关 MCP 教程和社区讨论。

MCP 渠道的内容重点：

- TraceMind MCP 是只读产品行为证据层。
- `projectKey` 只用于采集写入，MCP token 才能读取行为证据。
- agent 默认查语义事件，必要时复核原始行为。
- `tracemind.submit_feedback` 只在开发者明确要求时提交反馈。

### 4. 案例型内容

不要先写抽象产品介绍。早期内容应使用具体问题：

- "我用 Codex 查了一个 SaaS 的用户行为，发现新功能没人用不是入口问题，而是第二步配置卡住了。"
- "一行脚本加一个 MCP，让 Claude Code 直接回答用户为什么没有完成 onboarding。"
- "传统 analytics 需要先定义事件；TraceMind 先记录行为，再让 agent 追问。"

每篇内容都应该包含：

- 产品类型。
- 用户遇到的真实问题。
- TraceMind 如何接入。
- agent 问了什么。
- 发现了什么。
- 团队改了什么。
- 改完后如何验证。

### 5. Agency 和开发者工作室

找 AI app studio、独立开发团队、fractional CTO、Cursor agency。

合作方式：

- 提供一个默认接入模板。
- 帮他们把 TraceMind 放进交付清单。
- 让他们向客户交付 "上线后可被 agent 分析的行为证据"。
- 给每个工作室 3 个免费/低价试点项目。

## 30 天执行节奏

### 第 1 周：线索和访谈

- 建立 100 个高匹配线索。
- 每天发 10-15 个高质量私信。
- 约 10-15 个访谈。
- 访谈只问真实工作流，不推销完整功能。

访谈问题：

- 你们现在用哪些 AI coding 工具？
- 产品上线后，你们怎么知道用户有没有用新功能？
- 最近一次用户流失或转化下降，你们怎么查原因？
- 现在 analytics 的最大阻力是什么？
- 如果可以直接在 coding agent 里问用户行为，你最想问什么？

### 第 2 周：亲自接入

- 选择 5 个项目作为 design partner。
- 创始人亲自协助接入。
- 优先 Web 脚本和 `identify`，再配置 MCP token。
- 每个项目 24 小时内交付第一份诊断。

第一份诊断只回答 3 个问题：

- 用户在哪里流失？
- 哪个关键功能没人用？
- 最近转化或活跃变化是否对应某个页面、按钮、表单或流程？

### 第 3 周：推动产品修改

- 对每个项目给出 1-2 个最小产品修改建议。
- 让客户真的改一次，而不是只看报告。
- 改完后用 TraceMind 验证行为变化。

### 第 4 周：公开案例和 launch

- 形成 2 个匿名案例。
- 准备 Product Hunt、Hacker News、V2EX、即刻、X 的内容版本。
- 发布时不要强调 "dashboard"，强调 "coding agent can inspect real user behavior"。
- 把每次互动都加入线索表，持续跟进。

## 日常运营流程

每天 60-90 分钟即可开始：

1. 找 20 个新线索。
2. 筛掉没有真实产品、没有 AI coding 信号、没有联系方式的对象。
3. 写 10 条个性化外联。
4. 发布或回复 1 条内容。
5. 跟进昨天和前天的回复。
6. 更新线索状态和下一步动作。

线索状态：

- `new`: 新发现。
- `qualified`: 匹配 ICP。
- `contacted`: 已外联。
- `replied`: 有回复。
- `meeting`: 已约访谈。
- `pilot`: 进入接入。
- `active`: 已有真实数据。
- `case_study`: 可沉淀案例。
- `rejected`: 不匹配或无意愿。

## Codex、Computer Use 和浏览器工具如何协助

可以让 Codex 每天协助执行 GTM 任务，但需要把任务分成三类。

### 可自动完成

这些任务可以由 Codex 通过搜索、浏览器或 Computer Use 辅助完成：

- 搜索公开社区、目录、帖子、launch 页面和 GitHub 项目。
- 根据关键词整理潜在线索。
- 读取公开页面，判断是否符合 ICP。
- 为每个线索写个性化外联草稿。
- 为 X、V2EX、Product Hunt、Hacker News、即刻、微信群公告等渠道准备不同版本的帖子。
- 维护线索表、每日摘要、待跟进清单。
- 打开目标页面，准备好发帖或私信草稿。

### 需要用户确认后才能做

这些动作会对第三方产生外部影响，必须在动作前确认：

- 发送私信、邮件、评论、表单或社群帖子。
- 公开发布、修改或删除帖子。
- 点赞、关注、加入社群、申请账号。
- 上传文件或提交含有个人信息、客户信息、截图、账号信息的内容。
- 使用登录态在第三方网站执行任何代表用户表达态度或发送内容的动作。

确认时应明确：

- 要发到哪里。
- 要发给谁。
- 发送的正文是什么。
- 是否包含敏感信息或客户信息。

### 不应交给工具直接做

这些事情应由用户亲自处理或单独确认：

- 绕过安全警告、验证码、付费墙。
- 创建或保存密码、API key、OAuth 凭证。
- 涉及付款、订阅、退款、采购承诺的操作。
- 高风险账号权限、系统设置或云权限变更。

## 外联话术

中文私信：

```text

看你们产品已经上线，而且像是在用 AI coding 工作流。我想免费帮你们接入一次，24 小时内给一份“用户在哪里卡住”的诊断。如果没价值就不继续。
```

## 指标

早期不要只看注册数。优先看：

- `qualified_leads_per_day`: 每天新增合格线索。
- `reply_rate`: 外联回复率。
- `setup_started`: 开始接入项目数。
- `time_to_first_insight`: 从接入到第一条有用结论的时间。
- `agent_query_success`: 是否真的在 agent 中问出了结果。
- `insight_to_action`: 是否推动一次产品修改。
- `weekly_active_project`: 一周内是否再次查询。
- `second_project`: 同一团队是否接入第二个项目。
- `case_study_ready`: 是否可形成匿名案例。

## 运营原则

- 手动优先。早期不要追求群发自动化。
- 每条外联都要基于对方项目的具体观察。
- 不夸大自动分析能力，优先承诺 24 小时内协助拿到第一份诊断。
- 不发送客户隐私、截图、prompt、源码、token 或带 query 的完整 URL。
- 公开案例必须匿名化，除非客户明确授权。
- 如果对方不是 AI coding workflow 用户，先记录但不强推。
